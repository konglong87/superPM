## ✨ 核心设计原则

### 1. 轻量级 ✅

- **纯 Markdown 指令** - 无需编写代码
- **依赖 Claude 能力** - 利用 AI 理解与生成
- **每个 Skill 100-600 行** - 简洁高效

**示例**:
```markdown
## 步骤 2: 收集信息

使用 AskUserQuestion 逐个询问：

问题1: 请问您的产品名称是什么？
```

### 2. 一次一问 ✅

- ✅ 每次只问一个问题
- ✅ 提供 3-5 个选项
- ✅ 最后选项"其他（手动输入）"

**示例**:
```markdown
> 我要探索产品的哪个方向？
>
> A) 产品功能创新
> B) 用户增长方案
> C) 商业模式设计
> D) 用户体验优化
> E) 其他方向（请手动输入）
```

### 3. MCP 优先 ✅

- 优先使用 MCP Search（如 exa-mcp）
- MCP 不可用时降级 WebSearch
- 标注数据来源，便于验证

**数据源优先级**:
1. 艾瑞咨询、易观分析、QuestMobile
2. 七麦数据、蝉妈妈
3. 36氪、虎嗅
4. 工信部、网信办

### 4. Markdown 流转 ✅

- 前序 Skill → Write 输出 Markdown
- 后序 Skill → Read 读取文档
- 人类可读，可直接编辑

**流转示例**:
```
pm-demand (Write)
  → docs/01-需求调研/需求调研报告.md
  ↓
pm-clarify (Read)
  → 理解内容
  → Write
  → docs/01-需求调研/确认需求清单.md
```

### 5. 风险前置 ✅

- pm-demand 立即验证痛点
- pm-market 验证市场假设
- pm-mvp 验证 MVP 合理性

**示例**:
```markdown
**问题4（关键）**:
> 解决了什么用户痛点？
>
> 💡 提示：思考用户为什么需要这个产品
```

### 6. 兜底机制 ✅

- 文件缺失 → 提示执行前序 Skill
- 参数错误 → 引导用户输入
- 跨会话 → 检测进度并恢复

**示例**:
```markdown
⚠️ 未找到需求清单文件

您可以选择：
A) 执行 /pm-clarify 完成需求细化
B) 手动输入需求列表（快速模式）
```
